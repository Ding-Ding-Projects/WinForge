# CLAUDE.md — WinForge

Guidance for AI agents working in this repo. Keep it current.

## What this is
WinForge (a.k.a. 視窗調校) is an all-in-one, **fully bilingual (English + 繁體中文/粵語)** Windows 11 control center — 315 registered in-app modules plus a large Windows-tweak catalog — with a **hyper-realistic nuclear-reactor simulator** as its flagship. **.NET `net11.0-windows10.0.26100.0`, WinUI 3**, self-contained, unpackaged.

## Build / run / drive
- **Compile check:** `dotnet build WinForge.sln -c Debug -p:Platform=x64` (must finish with 0 errors; do not hard-code a warning count).
- **Run it:** a plain Debug build is framework-dependent and only shows a *"install .NET"* dialog here — you must **publish self-contained**:
  `dotnet publish WinForge.csproj -c Debug -r win-x64 --self-contained true -p:Platform=x64 -p:WindowsAppSDKSelfContained=true`
- **Easiest path — use the skill `.agents/skills/run-winforge/`** (SKILL.md + driver.ps1): builds-if-needed, launches any page via `--page <alias>`, screenshots the window. e.g.
  `powershell -ExecutionPolicy Bypass -File .agents/skills/run-winforge/driver.ps1 -Page reactor -Out shot.png`
- **Reactor/dependent headless tests:** `dotnet run --project tests/ReactorSim.Tests -c Debug` (**63/63**).
- The app is NOT a Start-menu app, so desktop/computer-use screenshot tools mask it — capture via the driver (DWM bounds + `CopyFromScreen`).

## Architecture & conventions (follow these)
- **Add a module = touch 4 places:** `Pages/<X>Module.xaml(.cs)` (class `<X>Module : Page`, namespace `WinForge.Pages`) + logic in `Services/<X>Service.cs`; then register in **(1)** `Services/ModuleRegistry.cs` (Tag `module.xxx`, En, Zh, Glyph, Keywords), **(2)** `MainWindow.xaml.cs` `MapType()` (tag→type), **(3)** `MainWindow.xaml.cs` `ApplyStartPage()` (deep-link aliases for `--page`), **(4)** `MainWindow.xaml` `NavigationViewItem`.
- **Language modes:** all user-facing strings use `Models/Core.cs` `LocalizedText(en, zh)` or `Services/Loc.cs` `Loc.I.Pick(en, zh)`. Cantonese (粵語), not Mandarin. The UI supports Bilingual, Cantonese, and English modes.
- **UI:** build **rich, control-based, animated UI** from real controls (ToggleSwitch/Slider/ComboBox/editors), mirroring `Pages/AwakeModule`. **`Controls/TweakCard` is FULLY REMOVED (zero tweak cards, definition files deleted)** — never reintroduce it; only historical comments mention it. To render a list of `TweakDefinition`s, use **`Controls/ControlRowList`** (`SetTweaks(IEnumerable<TweakDefinition>)` / `Clear()`) — it renders control rows (Action→Button awaiting `RunAsync`, Toggle→ToggleSwitch, Choice/RadioGroup→ComboBox/RadioButtons, Slider/Number→Slider/NumberBox, Info→refreshable TextBlock) with one persistent `InfoBar` + `EntranceThemeTransition`. For a bespoke module see the converted `Pages/RainmeterModule`/`Pages/CloudflareModule`; for catalog-driven pages see `Pages/SettingsHubModule` using `ControlRowList`. Use `Services/FileDialogs.cs` (never WinRT pickers). WebView2 is available. Pitfalls: classic `{Binding}` not `x:Bind` in DataTemplates; qualify `Windows.UI.Color`; `RadialGradientBrush.GradientStops` is read-only (Add, don't assign); use `ScrollViewer.*ScrollBarVisibility` attached props; a NAMED `Loc.I.LanguageChanged` handler unsubscribed on `Unloaded` (never an inline lambda — it leaks).
- **Pure managed C#** for module functionality; do not shell out to / launch the upstream program a module reimplements. Managed NuGet wrappers are fine (note them).
- **Apps that CAN'T be stuffed into a WinUI page → "Open full features" COMPANION apps, written in the upstream app's LITERAL language, still 100% WinForge.** Module pages KEEP their in-app GUI and gain a header button `Header.ActionContent = HeaderActions.FullFeaturesButton("<id>")` that opens a WinForge-branded popup: **web companions** (TS/JS — `WebApps/<id>/index.html`, served at `https://app.winforge/` by `Pages/CompanionWindows.cs` `WebAppWindow`; Monaco = VS Code's real engine auto-downloads once to `%LOCALAPPDATA%\WinForge\weblibs`, verifies a pinned SHA-512, and serves at `https://libs.winforge/`) and **native C++ companions** (`native/<id>/main.cpp` ships with the app; `Services/CompanionAppService.cs` finds g++/clang++/MSVC — **auto-installing `MartinStorsjo.LLVM-MinGW.UCRT` via winget when none exists** — compiles on demand behind `CompanionPrepWindow` with streamed output, caches by source hash in `%LOCALAPPDATA%\WinForge\native`, prebuilt fast-path `native\bin\<exe>`). C++ sources MUST compile with BOTH MinGW g++ and MSVC cl (contract in each file), and MinGW builds statically link their C++ runtime. Catalog: `CompanionAppService.All` (codeforge=Monaco editor, diagramforge=JS diagram studio, imageforge=C++ GDI+ image editor, audioforge=C++ audio editor). Bridge saves carry a `requestId`; host replies with matching `saveDone`; `language` / `theme` flow down. Companion preparation and interactive launches fail closed when WinForge itself is elevated; restart it normally first.
- **Apps that CAN'T be reimplemented ("stuffed") → the External App Launcher (not a reimplementation).** For big native GUI apps (Blender, GIMP, OBS, LibreOffice, VirtualBox, Wireshark, …) that must run in their **original native form as a popup**, add a **data-only** entry to `Catalog/ExternalApps.cs` (`ExternalAppSpec`: bilingual name/glyph, an ordered **winget dependency chain** — deps first, the app last — and exe-resolution hints: App Paths names / PATH stems / `%ProgramFiles%\…` candidates, `*`-glob allowed). Engine `Services/ExternalAppService.cs` resolves the real exe, installs the **whole chain touchlessly** (`InstallAllAsync`, streaming), and launches it only when WinForge is at normal integrity. UI: `Controls/AppLauncherCard.cs` (reusable — install via `InstallProgress`, explicit Launch, and a manual executable locator), shown as a **popup** by `Pages/AppLauncherWindow.cs`; the **App Launcher** hub (`Pages/AppLauncherModule`, tag `module.applauncher`, `--page applauncher`) lists them all. Wrapper modules cross-link via `Header.ActionContent = HeaderActions.NativeWindowButton("<id>")`. **Correct a winget id / install path ONLY in `ExternalApps.cs`**; adding an app = one manifest entry (no new page). This is the deliberate, opt-in exception to the "don't launch upstream programs" rule above.
- **Self-update path:** auto-update refuses elevated WinForge sessions. `WinForgeUpdater` verifies GitHub's SHA-256 asset digest, stages a copy of the single-file launcher outside the install folder, then exits; the helper waits for all WinForge binaries to unload, runs the per-user Inno installer with persistent logs, and relaunches.
- **Secrets:** DPAPI (`ProtectedData`) via the existing stores; never log secrets.

- Engine: `Services/ReactorSimService.cs` (+ `ReactorRps.cs`, `ReactorScenarios.cs`, `ReactorElectrical.cs`, `ReactorReactivityMeter.cs`). - **Safety invariants — keep these:** meltdown→real-shutdown is **default OFF** (abortable); reactor boots **held in MODE 5 cold shutdown** (operator must start it up); waste writes respect a disk free-space floor + a (default 50 GB) cap; all real-world side-effects are opt-in & reversible.
- **Foundational realism review P1–P5 is resolved and test-verified.** Backward-Euler kinetics is stable; a fresh fully-rodded core is subcritical (**−1018 pcm**); the corrected fuel/SG heat balance sustains a high-power equilibrium (**0.836→0.835 RTP, ~992.5 °C fuel, ~293.4 °C Tavg, 15.46 MPa RCS**) without emergency cooling, SCRAM, or meltdown; decay heat, xenon, saturated-pressurizer behavior, 2-of-4 RPS, OTΔT/OPΔT, SG low-low/AFW, turbine-trip cascade, 109% overpower, and 1/M are implemented. The full harness is **63/63 green**. Historical background: `docs/reactor-realism-review-001.md`; current evidence: `docs/wiki/Reactor-Test-Report.md`.

## Gotchas
- `audioeditor`, `lightswitch`, and `timelens` are fixed; if one captures blank via the driver, rerun with a longer `-WaitMs`.
- `--page <alias>` is reliable; bare `--reactor` can land on the Dashboard if a session was restored.
- Reactor restoring a stale/melted autosave → delete `%LOCALAPPDATA%\WinForge\state` (+ `…\session\tabs.json`).
- Git: branch + merge into `main`; never force-push; never touch unrelated personal remotes. Commit messages are bilingual.
